# S**3**

### what is S3
- s3 id a simple storage service, it is a amazon object storing service which can store the files, images, videos, PDF and others in the the cloud it can be structured and unstructured and semi-structured and it can connect to the other services as well and also retrieve any amount of data from anywhere
- Think of it like a giant online hard drive, but it's not really a "drive" in the traditional sense — there's no C:\ or folder structure underneath. It stores everything as objects inside buckets.

## **Core concepts**
### Bucket 
- It is a like a top level container , which is like the root folder where you store the objects
- Bucket name should be globally unique across all AWS not just the account name If some body had already a name like `my-photos` can't repeat
- Bucket name should be the in lower case and cant start with the underscore and space
- Buckets are created inside a specific AWS Region (e.g., ap-south-1 for Mumbai). This matters for latency (closer region = faster access) and compliance (some data must stay in a specific country).

### Object

An object is the actual file you store — could be an image, video, PDF, CSV, anything.
Each object has:

1. Key — this is the full path/name of the file (e.g., products/shoes/image1.jpg)
2. Value — the actual data/content
3. Metadata — info about the object (content type, size, custom tags, etc.)
4. Version ID — if versioning is enabled

### "Folders" aren't real

- This one surprises a lot of beginners. S3 is actually a flat structure — there are no real folders.
- When you see products/shoes/image1.jpg in the console, it looks like folders, but it's really just one long key name with / characters in it. The console just visually groups them to make it easier to browse.

### **Why people use S3**

1. Durability: AWS says S3 is designed for 99.999999999% durability (11 nines) — meaning your data is extremely unlikely to be lost, because it's automatically replicated across multiple facilities within a region.
2. Scalability: You don't provision storage size like a hard disk — it scales automatically. Store 1 file or 1 billion, no setup needed.
3. Pay-as-you-go: You pay for what you store, plus requests and data transfer — no upfront cost.
4. Accessible from anywhere: Every object gets a URL, and you can access it via Console, CLI, SDK, or REST API.

### **Real-World Example (building on your e-commerce doc)*
Let's say you're building that Amazon-like site again:

- Product images → stored in an S3 bucket
- Each image has a key like products/electronics/laptop123.jpg
- The bucket might be private by default (good practice), and your app fetches images through your backend or CloudFront, not directly exposing the bucket

### **A Few Basic Things to Know Before You Touch the Console**

1. Bucket names must be lowercase, no spaces, no underscores (only lowercase letters, numbers, hyphens, and dots) — because bucket names become part of a URL.
2. Public access is blocked by default now (AWS changed this a few years back for security) — you have to explicitly allow public access if you want it, which is a good safety default.
3. Everything you upload is private by default.
4. S3 is Region-scoped but globally accessible — meaning the bucket lives in one region, but you can access it from anywhere in the world (unless you restrict it).

## **Storage Classes**
### What are Storage Classes
- S3 doesn't store everything the same way — it offers different storage classes designed for different access patterns and cost needs
- The main tradeoff is always: how often you access the data vs how much you're okay paying
- Frequently accessed data = more expensive but instant access. Rarely accessed data = cheaper but slower or has retrieval costs

### The Storage Classes
**1. S3 Standard**
- Default storage class
- For data accessed frequently (more than once a month)
- Millisecond access, stored across ≥ 3 Availability Zones
- Most expensive, but no retrieval fees

**2. S3 Intelligent-Tiering**
- For data where you don't know the access pattern, or it keeps changing
- AWS automatically moves your object between tiers based on usage
- Has a small monitoring/auto-tiering fee for this automation

**3. S3 Standard-IA (Infrequent Access)**
- For data accessed about once a month
- Cheaper storage than Standard, but you pay a per-GB retrieval fee when you access it
- Minimum storage duration: 30 days (if you delete before that, you're still billed for 30 days)
- Minimum billable object size: 128 KB (even a 10 KB file gets billed as if it's 128 KB)

**4. S3 One Zone-IA**
- Same as Standard-IA, but stored in only 1 Availability Zone instead of 3+
- Cheaper, but riskier — if that one zone has an issue, you could lose the data
- Good for recreatable data (like a copy of something you already have elsewhere)

**5. S3 Glacier Instant Retrieval**
- Archive storage, but still gives millisecond access
- Meant for data accessed about once a quarter
- Minimum storage duration: 90 days

**6. S3 Glacier Flexible Retrieval**
- Archive storage for data accessed about once a year
- Retrieval takes minutes to hours (not instant)
- Minimum storage duration: 90 days

**7. S3 Glacier Deep Archive**
- Cheapest storage class AWS offers
- For data accessed less than once a year
- Retrieval takes hours
- Minimum storage duration: 180 days

**8. Reduced Redundancy (legacy)**
- Older storage class, not recommended anymore
- Standard is actually more cost-effective now, so this one is mostly deprecated

### Quick Comparison Table
| Storage Class | Access Frequency | Min Duration | Min Billable Size | Retrieval Fee |
|---|---|---|---|---|
| Standard | Frequent | - | - | No |
| Intelligent-Tiering | Unknown/changing | - | - | No (monitoring fee) |
| Standard-IA | ~Monthly | 30 days | 128 KB | Yes |
| One Zone-IA | ~Monthly (1 AZ) | 30 days | 128 KB | Yes |
| Glacier Instant Retrieval | ~Quarterly | 90 days | 128 KB | Yes |
| Glacier Flexible Retrieval | ~Yearly | 90 days | - | Yes |
| Glacier Deep Archive | Rare (<1/year) | 180 days | - | Yes |

### Changing Storage Class
- You can manually change an object's storage class from the console (Object → Edit storage class)
- Important: this actually creates a copy of the object with a new last-modified date — it's not a simple in-place edit
- Doing this manually for many objects isn't practical — that's what Lifecycle Rules are for

### Lifecycle Rules
- A Lifecycle Rule automates storage class transitions and expiration based on object age
- Example: "Move objects to Standard-IA after 30 days → move to Glacier after 90 days → delete after 365 days"
- This is the real-world way companies manage cost — they don't manually reclassify files one by one

### Real-World Example
Imagine a company storing user-uploaded documents:
- Day 0–30: Files sit in Standard (people might still access them regularly)
- Day 30–90: Files move to Standard-IA (access has slowed down)
- Day 90+: Files move to Glacier Flexible Retrieval (rarely touched, kept for compliance/backup only)
- Day 365+: Files get deleted automatically via an expiration rule

This is set up once as a Lifecycle Rule, and S3 handles the rest — no manual work needed.

## **Permissions & Security**
### Why this matters
- This is where most real-world S3 mistakes happen — misconfigured permissions are one of the most common causes of data breaches
- By default, everything in S3 is private. You have to explicitly grant access — nothing is public unless you make it public
- There are multiple layers of access control, and they can overlap, so it's important to know which one is doing what

### Bucket Policies
- A JSON-based policy attached directly to a bucket
- Defines who (which AWS account, user, or role) can do what (read, write, delete) on that bucket or its objects
- Resource-level control — it lives on the bucket, not on the user
- Example use case: "Allow public read access to only the `images/` folder in this bucket"

### IAM Policies
- Also JSON-based, but attached to an IAM user, group, or role — not the bucket
- Defines what that user/role is allowed to do across AWS, including S3
- Difference from Bucket Policy: IAM Policy = permission attached to the *person/role*, Bucket Policy = permission attached to the *resource*
- Both can be used together, and both are checked — access is only granted if nothing explicitly denies it

### ACLs (Access Control Lists)
- The oldest, legacy way of managing access — grants permissions at the individual object or bucket level
- AWS now recommends avoiding ACLs in favor of Bucket Policies and IAM Policies
- Most modern buckets have ACLs disabled entirely

### Block Public Access
- A setting AWS added to prevent accidental public exposure of buckets
- Enabled by default on all new buckets
- Even if a Bucket Policy tries to allow public access, this setting can override and block it unless turned off intentionally
- Best practice: keep this on unless you have a clear, specific reason to host public content (like a static website)

### Presigned URLs
- A temporary, time-limited URL that grants access to a private object without making the whole bucket public
- Useful when you want to let someone download or upload a specific file for a short window (e.g., 15 minutes)
- Generated using AWS CLI/SDK — the URL contains a signature that proves it was authorized
- Example use case: letting a user download their invoice PDF without exposing the entire bucket

### Encryption
S3 encrypts data in different ways depending on who manages the encryption key:

**1. SSE-S3 (Server-Side Encryption with S3-managed keys)**
- AWS manages the encryption key entirely
- Simplest option, encryption happens automatically

**2. SSE-KMS (Server-Side Encryption with AWS KMS keys)**
- Uses AWS Key Management Service (KMS) to manage the key
- Gives more control — you can set who's allowed to use the key, and get an audit trail of key usage

**3. SSE-C (Server-Side Encryption with Customer-provided keys)**
- You provide and manage your own encryption key
- AWS encrypts/decrypts using your key but doesn't store it

**4. Client-Side Encryption**
- You encrypt the data yourself before uploading it
- AWS never sees the unencrypted data at all

### Access Logging
- S3 Server Access Logs record every request made to a bucket (who accessed what, when, from where)
- Useful for security audits and understanding usage patterns
- Logs are usually stored in a separate S3 bucket

### Real-World Example
Imagine a company storing customer ID documents in S3:
- Bucket has **Block Public Access enabled** — nothing is public
- **IAM Policy** on the backend server's role allows it to read/write documents
- **Bucket Policy** explicitly denies access from any account other than the company's own
- Documents are encrypted using **SSE-KMS** so only authorized roles can decrypt them
- When a customer needs to download their own document, the app generates a **Presigned URL** valid for 5 minutes
- **Access logging** is enabled so the security team can audit who accessed what

This layered approach — private by default, explicit grants only where needed, encryption at rest, and short-lived access — is the standard real-world pattern.

## **Versioning & Object Lock**
### What is Versioning
- By default, if you upload a file with the same name (key) as an existing one, S3 overwrites it — the old file is gone permanently
- Versioning changes this: when enabled, S3 keeps every version of an object instead of overwriting it
- Each version gets a unique Version ID
- This protects against accidental overwrites and accidental deletes

### How Versioning is Disabled by Default
- New buckets have versioning **disabled** by default
- AWS actually warns you about this in the console — it recommends enabling it to protect against unintended user actions and application failures
- Once enabled, versioning **cannot be fully turned off** — it can only be **suspended** (existing versions stay, but new overwrites won't create new versions until re-enabled)

### How Deletes Work With Versioning
- Without versioning: deleting an object removes it permanently
- With versioning enabled: deleting an object doesn't actually delete it — instead, S3 adds a **Delete Marker**
- The Delete Marker just hides the object from normal view (like a soft delete)
- The actual object data is still there in a previous version — you can restore it by removing the Delete Marker
- To permanently delete a specific version, you must delete that exact Version ID

### MFA Delete
- An extra layer of protection on top of versioning
- Requires multi-factor authentication (MFA) before someone can permanently delete an object version or change versioning state
- Mainly used for critical/compliance data where accidental or malicious deletion needs to be extra hard to do

### Replication
**Cross-Region Replication (CRR)**
- Automatically copies objects from a bucket in one region to a bucket in a different region
- Useful for disaster recovery, compliance (keeping a copy in another country), or reducing latency for users in another region

**Same-Region Replication (SRR)**
- Same concept, but copies within the same region
- Useful for keeping a separate copy for compliance, logging, or aggregating data from multiple buckets into one

- Both require versioning to be enabled on the source bucket

### Object Lock (WORM)
- WORM = Write Once, Read Many
- Prevents an object from being deleted or overwritten for a fixed amount of time — or indefinitely
- Only works on buckets that have **versioning enabled**
- Used for compliance situations where data must be tamper-proof (e.g., financial records, legal documents)

**Two modes of Object Lock:**
1. **Governance mode** — most users can't delete/overwrite the object, but users with special permissions can override the lock
2. **Compliance mode** — nobody, including the root account, can delete or overwrite the object until the lock period expires

### Real-World Example
Imagine a company storing signed contracts:
- **Versioning is enabled** — so if someone accidentally re-uploads a contract with the same name, the old signed version isn't lost
- **Object Lock (Compliance mode)** is applied for 7 years — nobody, not even an admin, can delete these files during that period
- **Cross-Region Replication** copies contracts to a bucket in another region for disaster recovery
- If someone deletes a contract by mistake, S3 only adds a **Delete Marker** — the actual signed contract is still recoverable

This combination — versioning + Object Lock + replication — is exactly how companies handle legally sensitive or compliance-critical documents in S3.
